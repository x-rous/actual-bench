## Codebase Quick Reference

Before implementing anything, read these key facts about the architecture:

**API layer** (`src/lib/api/`)
All API functions live here. Every function imports `apiRequest` from `./client` and normalizes API snake_case types ‚Üí internal camelCase types. Pattern: `getXxx`, `createXxx`, `updateXxx`, `deleteXxx`. Never use `ApiXxx` types outside this folder.

**Proxy** (`src/app/api/proxy/route.ts`)
All browser fetches route through here ‚Äî no direct browser‚ÜíAPI calls. The proxy serializes requests per-server via a `serverQueueTails` Map to prevent concurrent budget open/close races in actual-http-api. Budget-scoped paths are automatically prefixed: `path = "/accounts"` ‚Üí `GET /v1/budgets/{budgetSyncId}/accounts`. The proxy currently calls `upstreamResponse.json()` on all 200 responses ‚Äî binary responses (e.g. budget export ZIP) require special handling.

**Entity types** (`src/types/entities.ts`)
Internal normalized types. `Schedule` and `RecurConfig` are fully defined. `Tag` exists but is missing `color` and `description` fields (added in Actual Budget v26.3.0). `Transaction` type does not exist yet.

**Staged store** (`src/store/staged.ts`)
All mutations go through `stageNew`, `stageUpdate`, `stageDelete`. Data is not sent to the server until `Save` is clicked. The store holds `accounts`, `payees`, `categoryGroups`, `categories`, `rules`, `schedules` slices. All entity slices including rules are prefetched on every page via `usePreloadEntities()` in AppShell ‚Äî `stagedRules` is populated as soon as the app loads.

**Connection** (`src/store/connection.ts`)
`ConnectionInstance` type: `{ id, label, baseUrl, apiKey, budgetSyncId, encryptionPassword? }`. Persisted to sessionStorage via Zustand persist middleware.

**Feature pages** (`src/features/`)
Each feature follows the same structure: `components/XxxView.tsx` (page shell + toolbar), `components/XxxTable.tsx` (table + inline edit + dialogs), `hooks/useXxx.ts` (React Query fetch), `hooks/useXxxSave.ts` (save logic), `csv/xxxCsvExport.ts`, `csv/xxxCsvImport.ts`, `schemas/xxx.schema.ts`. Follow this pattern exactly for new features.

**React Query** ‚Äî all data fetching uses `@tanstack/react-query`. See `src/lib/queryClient.ts` for the shared client. Use `useQuery` for reads, invalidate on save. Global defaults: `staleTime: Infinity`, `gcTime: Infinity`, `refetchOnWindowFocus: false`, `refetchOnReconnect: false` ‚Äî do not add per-hook overrides unless intentionally deviating (e.g. time-sensitive data like balances or notes).

**UI components** (`src/components/ui/`) ‚Äî shared: `Button`, `Input`, `Label`, `Badge`, `Dialog`, `Sheet`, `DropdownMenu`. Use these; do not install new component libraries.

**`generateId()`** ‚Äî always use `import { generateId } from "@/lib/uuid"` for new IDs. Never use `crypto.randomUUID()` directly (fails on HTTP).

### RD-006 ‚Äî Budget Export Download

| | |
|---|---|
| **Priority** | Medium |
| **Effort** | S |
| **Status** | pending |
| **Depends on** | Proxy binary response patch |

**What:** A one-click button that downloads a complete backup ZIP of the active budget. The ZIP contains `db.sqlite` + `metadata.json`.

**API endpoint:**
```
GET /budgets/{id}/export
Response: application/zip binary stream
```

**Critical proxy issue:** The proxy (`src/app/api/proxy/route.ts`) currently calls `upstreamResponse.json()` on all successful responses (line 126). This will fail for the binary ZIP response. The proxy needs to detect `Content-Type: application/zip` and return the raw binary instead.

**Proxy patch required** in `upstreamFetch()`:
```ts
// After the ok check, before the json() call:
const contentType = upstreamResponse.headers.get("content-type") ?? "";
if (contentType.includes("zip") || contentType.includes("octet-stream")) {
  const blob = await upstreamResponse.arrayBuffer();
  return new NextResponse(blob, {
    status: 200,
    headers: { "Content-Type": contentType, "Content-Disposition": "attachment; filename=\"budget-export.zip\"" },
  });
}
```

**Client-side** in `src/lib/api/client.ts` ‚Äî add a separate `downloadFile(connection, path)` function that uses `fetch` directly (not via `apiRequest`) and creates a browser download via `URL.createObjectURL`.

**Files to update:**
- `src/app/api/proxy/route.ts` ‚Äî patch `upstreamFetch` for binary responses
- `src/lib/api/client.ts` ‚Äî add `downloadFile` helper
- `src/components/layout/TopBar.tsx` or connection panel ‚Äî add "Export Budget" button

**Implementation notes:**
- Place the export button in the TopBar's connection dropdown (the one showing the active instance label). Add it as a menu item: `Download Backup`.
- The export can take a few seconds ‚Äî show a spinner on the button during the request.
- The filename should include the budget label and date: `budget-{label}-{YYYY-MM-DD}.zip`.

### RD-017 ‚Äî Transaction Count Warnings on Delete

| | |
|---|---|
| **Priority** | Medium |
| **Effort** | S |
| **Status** | pending |
| **Depends on** | RD-016 (confirmation dialogs must exist first); RD-007 shares the `runQuery` utility |

**What:** Before deleting a payee or category, show how many transactions reference it. Deleting does not remove transactions ‚Äî it unlinks them (sets `payee_id` / `category_id` to null). Users should be warned so they understand the data impact.

**Analysis findings:**
- Actual Budget does not delete transactions when a payee or category is deleted ‚Äî it nulls the reference.
- The warning copy should be precise: *"47 transactions will be unlinked (not deleted). Their payee will be cleared."*
- Transaction counts can be fetched in a single `run-query` call per entity type, grouped by ID.

**API endpoint:**
```
POST /budgets/{id}/run-query
Body for payee counts:
{
  "ActualQLquery": {
    "table": "transactions",
    "select": ["payee_id"],
    "groupBy": ["payee_id"],
    "calculate": [{ "name": "count", "func": "count", "field": "id" }]
  }
}
Response: { data: [{ payee_id: "abc123", count: 47 }, ...] }

Body for category counts:
{
  "ActualQLquery": {
    "table": "transactions",
    "select": ["category_id"],
    "groupBy": ["category_id"],
    "calculate": [{ "name": "count", "func": "count", "field": "id" }]
  }
}
```

**New API function:**
```
src/lib/api/query.ts  (also used by RD-007)
  runQuery(connection, query: ActualQLQuery): Promise<Record<string, unknown>[]>
```

**New hooks:**
```
src/hooks/usePayeeTransactionCounts.ts
  ‚Äî calls runQuery with payee groupBy query
  ‚Äî returns Map<payeeId, count>
  ‚Äî useQuery with enabled: false initially; call refetch() on first delete
  ‚Äî staleTime: 120_000 (2 min ‚Äî intentional override of the global Infinity default)

src/hooks/useCategoryTransactionCounts.ts
  ‚Äî same pattern for categories
```

**Fetch strategy ‚Äî lazy batch (recommended):**
- On the Payees or Categories page, do NOT fetch on mount.
- On the first delete action (single or bulk), call `refetch()` to trigger the query.
- Cache the result for 2 minutes in React Query.
- Show a brief spinner in the confirm button while the counts are loading.
- Subsequent deletes in the same session use the cached result instantly.

**Files to update:**
- `src/features/payees/components/PayeesTable.tsx` ‚Äî add `usePayeeTransactionCounts`; augment both single-delete and bulk-delete dialogs with the count
- `src/features/categories/components/CategoriesTable.tsx` ‚Äî add `useCategoryTransactionCounts`; augment single, group, and bulk delete dialogs

**Warning message format:**
- If count is 0: no additional warning (don't clutter the dialog)
- If count > 0: append to confirmation dialog: *"‚öÝ {N} transaction(s) referencing this {payee/category} will be unlinked (not deleted)."*

**Important constraint:** The `run-query` endpoint is subject to the same per-server queue in the proxy. Do not fire the query on every keystroke or mouse movement ‚Äî only on explicit delete intent (button click).

## Analysis Notes for Future Implementation Sessions

The following findings from code analysis should be kept in mind when working on any of the above items:

**`stagedRules` reliability:** Rules are prefetched on every page via `usePreloadEntities()` in AppShell ‚Äî `stagedRules` is populated as soon as the app loads and does not require the Rules page to be visited first. For dependency count checks (e.g. before deleting a payee), scanning `stagedRules` is reliable. Server-side API calls (`GET /payees/:id/rules`, or a `run-query`) remain a valid fallback for fresh counts after a save.

**Proxy queue awareness:** All `run-query` calls and API fetches go through the per-server queue in `route.ts`. This means a `run-query` fired during a delete confirmation will be queued behind any in-flight save operations. This is correct behavior ‚Äî do not bypass the queue.

**`Tag` type is stale:** `src/types/entities.ts` has a comment saying *"Tags are not implemented in the current Actual HTTP API"* ‚Äî this is wrong as of v26.3.0. Update the comment and add the `color` and `description` fields when implementing RD-002.

**`ACTION_FIELDS` is incomplete:** Missing `account`, `deleted`, and `link-schedule` action types. RD-008 adds `deleted` and `account`. `link-schedule` waits for RD-001.

**`Schedule.date` duality:** `date` is `string | RecurConfig`. When reading from the API, check `typeof date === "string"` to distinguish one-time vs recurring. When writing, the API expects either a date string or a full `RecurConfig` object ‚Äî do not send both.

**Budget export binary response:** The proxy's `upstreamFetch` function calls `upstreamResponse.json()` unconditionally on success responses. RD-006 MUST patch this before the export endpoint will work.

**Connection `budgetSyncId` in proxy:** The proxy automatically constructs the full path: if `connection.budgetSyncId` is present, it prepends `/v1/budgets/{budgetSyncId}` to the `path` argument. So `path = "/run-query"` correctly becomes `POST /v1/budgets/{id}/run-query`. No manual path construction needed in API functions.

**`generateId()` ‚Äî always use it:** Never use `crypto.randomUUID()` directly. The `generateId()` wrapper in `src/lib/uuid.ts` falls back to `Math.random` on HTTP (issue #13, fixed). All new code must use `generateId()`.

### RD-023 ‚Äî Rule Diagnostics / Linting

| | |
|---|---|
| **Priority** | High |
| **Effort** | M |
| **Status** | pending |
| **Depends on** | RD-007 |

**What:** A diagnostics tool that scans rules and highlights likely issues, risky patterns, and cleanup opportunities. This should extend the rule-management experience without recreating Actual Budget's rule engine.

**Why:**
- Strong value-add for power users
- Helps keep large rule sets understandable and maintainable
- Fits actual-bench's role as the place for advanced administration and analysis
- Builds on existing rule work already delivered (templates, regex hints, more action types, grouped categories)

**Scope v1 checks:**
- Duplicate or near-duplicate rules
- Rules referencing missing / deleted entities
- Rules with empty or no-op actions
- Rules with impossible or contradictory conditions
- Rules that appear shadowed by earlier rules in the same stage
- Rules with suspicious broad match criteria (for example `contains` with very short values)
- Rules with deprecated / unsupported field or op combinations if any are detected

**Nice-to-have checks (v2):**
- Heuristic overlap detection between rules
- Identify rules that could be merged
- Warn when a rule's actions likely override another rule's actions later in the same processing flow
- Suggest use of templates or grouped categories where it simplifies maintenance

**Files to create:**
```text
src/features/rule-diagnostics/
  components/RuleDiagnosticsView.tsx
  components/RuleDiagnosticsTable.tsx
  components/DiagnosticSummaryCards.tsx
  lib/ruleDiagnostics.ts
  lib/ruleLintRules.ts
  types.ts
```

**Files to update:**
- `src/app/(app)/rules/page.tsx` or `src/features/rules/components/RulesView.tsx` ‚Äî add entry point to open diagnostics
- `src/components/layout/Sidebar.tsx` ‚Äî optional direct nav item if the feature becomes substantial enough

**Implementation approach:**
- Start with client-side analysis using the already-loaded normalized rules from the staged store
- Treat diagnostics as advisory only ‚Äî never auto-modify rules in v1
- Keep findings deterministic and explainable; every warning should say why it was raised
- Group results by severity: `error`, `warning`, `info`

**Output model example:**
```ts
type RuleDiagnostic = {
  ruleId: string;
  severity: "error" | "warning" | "info";
  code: string;
  title: string;
  message: string;
  details?: string[];
};
```

**UI notes:**
- Provide summary cards at the top: total issues, errors, warnings, infos
- Table columns: severity, rule, issue, suggested action
- Clicking a diagnostic should open or focus the related rule in the Rules page where possible
- Allow filters by severity and issue type

**Design guardrails:**
- Do not attempt to reimplement Actual's full evaluation engine
- Prefer transparent heuristics over overly clever scoring
- False positives are acceptable if the explanation is clear and the severity is conservative

**Possible future integration:**
- Show a small warning badge on the Rules nav item when diagnostics detect errors
- Link to RD-021 Usage Inspector for missing-entity and reference-related issues
- Offer export to CSV / JSON later if users find the diagnostics useful
